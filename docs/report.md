# 3D-реконструкция лица по 1-3 фото: проектный отчет

## 1. Цель и контекст проекта

Цель проекта — построить воспроизводимый пайплайн, который по 1-3 фотографиям со
смартфона формирует web-ready 3D-модель головы: стабильный mesh, управляемую
топологию, корректную UV-текстуру, набор масок/карт материала и удобный
браузерный preview для быстрой проверки результата.

Ключевое ограничение входа — малое число изображений. Это сразу отсеивает
классическую фотограмметрию как основной путь: ей обычно нужно больше ракурсов,
более контролируемая съемка, стабильное освещение и неизменное выражение лица.
С другой стороны, полностью генеративные методы могут достраивать недостающие
части головы, но несут риск потери идентичности. Поэтому наиболее практичная
стратегия для проекта — гибридная: использовать параметрическую модель лица как
канонический каркас, а нейросетевые и texture-authoring инструменты подключать
для реконструкции формы, детализации, текстур и масок.

Проект развивается вокруг связки:

```text
input photos
  -> quality gate
  -> face detection and alignment
  -> DECA/FLAME reconstruction
  -> semantic masks and UV texture baking
  -> material maps
  -> GLB export
  -> Three.js QA
```

Главный инженерный приоритет — не максимальная плотность сетки сама по себе, а
согласованность геометрии, UV-развертки, текстур, масок и материала. Именно эта
согласованность определяет, будет ли результат пригоден для реального просмотра,
дальнейшей обработки и интеграции в браузерную 3D-среду.

## 2. Классы методов и их применимость

### 2.1. 3D Morphable Models и FLAME

3D Morphable Models описывают лицо через параметры формы, выражения и позы.
FLAME — один из самых полезных представителей этого подхода: он дает стабильную
топологию головы, управляемые expression/pose параметры и предсказуемый mesh.

Сильная сторона FLAME — не фотореализм из коробки, а инженерная управляемость.
Если проекту нужен один канонический mesh, который можно анимировать, сравнивать
между ракурсами, экспортировать в GLB и использовать как основу для texture
baking, FLAME хорошо подходит как базовый слой.

Ограничение FLAME состоит в том, что сама модель не решает задачу
персонализированной текстуры, волос, кожи, пор, аксессуаров и сложных
окклюзий. Эти элементы должны появляться на следующих этапах: через DECA,
semantic masks, UV-проекцию, texture cleanup и material maps.

### 2.2. DECA и EMOCA

DECA восстанавливает параметры лица, позу, выражение, освещение, альбедо и
детальные смещения по одному изображению. Это делает его удобным baseline для
быстрого image-to-mesh результата. Важная особенность DECA — разделение
coarse-формы и detail/displacement слоя. На практике это позволяет получить
легкий базовый mesh и отдельно использовать карты деталей для normal/bump или
dense export.

EMOCA развивает идею DECA с акцентом на эмоциональные выражения и более
устойчивую мимику. Для проекта EMOCA логично рассматривать не как замену всего
пайплайна, а как альтернативный initializer или ветку для случаев, где выражение
лица сильно влияет на результат.

### 2.3. MICA и метрическая форма

MICA полезна там, где важна не просто визуально похожая форма, а более
метрически корректный identity shape. Для multi-view сценария MICA можно
использовать как дополнительную стабилизацию формы: фронтальный и профильные
кадры могут давать разные оценки, а общий identity shape должен быть один.

### 2.4. Фотограмметрия, NeRF и 3D Gaussian Splatting

Фотограмметрия и SfM/MVS дают сильный результат при большом наборе кадров, но
плохо совпадают с ограничением 1-3 фото. NeRF и 3D Gaussian Splatting сильны в
novel-view rendering и фотореалистичности, но не всегда дают чистый mesh со
стабильной топологией. Для лица, которое нужно экспортировать, проверять,
маскировать и использовать в browser/WebGL pipeline, они скорее являются
R&D-направлением вокруг baseline, а не первым слоем архитектуры.

### 2.5. ComfyUI, segmentation и DCC-инструменты

ComfyUI, SAM/BiSeNet-style segmentation, depth/normal estimation, Blender,
Krita, GIMP, ArmorPaint и похожие инструменты важны как production control
layer. Они не отменяют DECA/FLAME, а усиливают pipeline вокруг них:

- выделяют кожу, волосы, шею, уши, губы, глаза и силуэт;
- помогают строить alpha/ID masks;
- улучшают texture cleanup;
- позволяют переносить детали в normal/height/roughness/cavity maps;
- дают ручной или semi-manual контроль там, где нейросеть ошибается.

## 3. Выбранная стратегия

Рациональная стратегия для проекта — держать FLAME/DECA как канонический слой
геометрии и постепенно строить вокруг него texture/mask pipeline.

Базовый контур:

```text
1-3 photos
  -> quality gate
  -> face/head segmentation
  -> DECA/EMOCA/MICA initialization
  -> shared FLAME identity fitting
  -> UV texture baking
  -> mask/material map generation
  -> GLB export
  -> browser QA
```

Такой подход дает несколько практических преимуществ:

- стабильная топология остается контролируемой;
- можно сравнивать coarse/detail варианты без смены всей архитектуры;
- текстуры и маски становятся отдельными проверяемыми артефактами;
- browser viewer используется как QA-инструмент, а не только как витрина;
- можно добавлять новые модели и segmentation-ветки без переписывания всего
  проекта.

## 4. Практический baseline проекта

В проекте уже реализован рабочий контур:

- Python CLI для генерации и реконструкции мешей;
- standalone `Flame2023Decoder` на PyTorch;
- DECA GPU path с CUDA и standard rasterizer;
- экспорт OBJ/GLB/PLY;
- mesh diagnostics для bounds, connected components, degenerate faces и UV;
- Three.js viewer с переключением ассетов, wireframe и скачиванием GLB;
- документация по архитектуре, DECA/FLAME интеграции и multi-view выводам;
- тесты на декодер, экспорт, diagnostics и CLI.

В viewer сейчас сравниваются несколько типов ассетов:

- `DECA result` — coarse 5k mesh из трех проекций;
- `DECA baked detail` — тот же легкий mesh с baked normal/bump maps;
- `DECA detail` — плотная DECA detail-геометрия;
- `FLAME textured` / `FLAME mask` / `FLAME neutral` — стабильный FLAME baseline;
- `FLAME detail textured` / `FLAME detail` — subdivided FLAME topology.

Эта практическая часть важна не только как демонстрация работы кода. Она
позволила увидеть реальные ограничения: где текстура темнеет из-за материала,
где dense mesh раскрывает артефакты, где UV-пространства не совпадают, а где
лучше оставить легкую геометрию и переносить детали в карты материала.

## 5. Выводы после экспериментов

### 5.1. Больше полигонов не равно лучше

Плотная сетка показывает микрорельеф и форму лучше, но одновременно раскрывает
ошибки генерации, UV-швы, размытие и рассинхрон текстуры. Coarse mesh на
5k-10k вершин часто выглядит чище, потому что сглаживает артефакты. Это не
значит, что dense mesh не нужен; это значит, что он отвечает на другой вопрос.

Coarse mesh полезен для:

- real-time preview;
- быстрой загрузки в браузере;
- стабильного сравнения формы;
- переноса микродеталей в normal/detail normal/height maps.

Dense mesh полезен для:

- анализа displacement;
- проверки микрогеометрии;
- offline export;
- сравнения того, что именно восстанавливает DECA detail branch.

Практический компромисс для real-time:

```text
coarse mesh + baseColor/albedo + normal map + optional height/bump map
```

Этот подход ближе к игровому и production-пайплайну, где детализация лица редко
хранится только в геометрии.

### 5.2. Маски нужно вести через 2D/UV pipeline

Эксперимент `DECA + FLAME mask`, где FLAME texture-space mask запекалась прямо в
vertex colors DECA mesh, удален из демо. Он показал рассинхрон пространств:
маска визуально плывет по лицу, потому что FLAME UV/texture assumptions нельзя
механически перенести на DECA preview geometry.

Правильное направление:

```text
semantic image masks
  -> visibility-aware projection
  -> target UV texture
  -> material maps on the selected mesh
```

Если маска создана в FLAME UV, а целевой ассет использует другой UV layout,
нужен явный texture bake/remap в целевое UV-пространство. Простое увеличение
числа faces или перенос цвета в вершины не решает проблему. Вершины дают только
грубую интерполяцию по треугольникам, а для лица нужны 2D-карты: skin, lips,
eyes, eyelids, ears, neck, hairline, beard/stubble, cavity, roughness, SSS и
другие зоны.

### 5.3. Multi-view — это не склейка мешей

DECA на трех проекциях дает три независимые оценки параметров. Все они
раскладываются в FLAME-подобную топологию, поэтому геометрию не нужно
склеивать вручную. Нужно решать совместную оптимизацию:

- shared identity shape;
- per-view camera, pose, expression, lighting;
- visibility masks;
- landmark, photometric и silhouette losses;
- UV texture fusion с весами по видимости и углу обзора.

Минимальный полезный multi-view pipeline:

```text
images[3]
  -> DECA encode per image
  -> robust mean / optimization for shared identity shape
  -> per-view pose/camera/expression/light
  -> semantic masks and visibility
  -> UV texture fusion
  -> one canonical mesh
```

Ручная склейка мешей создала бы швы, двойные поверхности и разные локальные
искажения. Параметрическое согласование сохраняет чистую топологию.

## 6. Риски и контроль качества

Главный риск проекта — domain gap между обучающими данными моделей и реальными
фото пользователя. На практике ошибки появляются из-за:

- бокового или контрового света;
- профиля вместо фронтального кадра;
- сильной мимики;
- очков, волос, головных уборов и других окклюзий;
- JPEG-артефактов, motion blur и низкого разрешения;
- несоответствия демографии и условий съемки обучающим наборам;
- несовместимых версий FLAME/DECA assets;
- неправильного UV layout или перепутанного channel order текстуры.

Для контроля качества нужен не один итоговый скриншот, а набор проверок:

| Уровень | Что проверять | Практический сигнал |
|---|---|---|
| Input QA | резкость, экспозиция, размер лица, поза | просить переснять фото до инференса |
| Geometry | bounds, topology, degenerate faces | diagnostics перед экспортом |
| Identity | стабильность shape между ракурсами | shared shape consistency |
| Texture | UV seams, channel order, projection drift | отдельный texture preview |
| Masks | semantic zones, hair/skin/neck edges | 2D QA + UV QA |
| Material | roughness, normals, bump, SSS | browser/DCC preview |
| Export | GLB/OBJ загрузка, scale, normals | Three.js smoke-test |

## 7. Лицензированные ассеты и воспроизводимость

Репозиторий не должен распространять закрытые или лицензируемые model assets.
После клонирования можно запускать тесты, mock-режим, читать документацию и
открывать подготовленный viewer, но полноценную генерацию с нуля нельзя
воспроизвести без самостоятельной загрузки моделей.

Для практического запуска нужны:

- DECA checkpoint `deca_model.tar`;
- FLAME2020 `generic_model.pkl` для DECA;
- FLAME2023 / FLAME2023 Open `.pkl` для standalone generation;
- `FLAME_texture.npz`;
- сопутствующие UV, mask, landmark и template assets.

Все эти файлы нужно получать из официальных источников после регистрации,
принятия лицензий и соблюдения правил использования. Это важная часть
инженерной воспроизводимости: код может быть открыт, но данные и модели имеют
собственные юридические ограничения.

## 8. Роль дополнительных исследований

В документацию добавлены два практических исследования:

- `docs/research/08-comfyui-masks-meshes.md` — как использовать ComfyUI,
  segmentation, depth/normal estimation, image-to-3D ветки и ручную чистку масок
  вокруг DECA/FLAME.
- `docs/research/09-game-face-textures-masks.md` — как игровые пайплайны строят
  лицо через набор PBR-карт: base color, normal/detail normal, roughness,
  cavity/AO, SSS, ID masks, wrinkle maps и channel packing.

Оба документа усиливают главный вывод: качество финальной головы зависит от
системы карт и масок, а не только от нейросетевой реконструкции меша. В игровой
и real-time среде лицо почти всегда является суммой нескольких карт материала,
а не одной “текстуры лица”.

## 9. Roadmap

Ближайшие задачи:

1. Реализовать явный UV texture/mask bake в целевую развертку выбранного mesh.
2. Добавить semantic face parsing для кожи, губ, глаз, век, ушей, волос, шеи и
   силуэта.
3. Проверить multi-view texture fusion через visibility/angle weights.
4. Разделить viewer-режимы: geometry, texture, normal, mask, wireframe.
5. Продолжить сравнение coarse/detail вариантов как разных режимов качества.
6. Подготовить API-слой для загрузки фото и серверного инференса.
7. Добавить DCC-oriented export для Blender/ArmorPaint cleanup.

Среднесрочное направление:

- использовать DECA/EMOCA/MICA как initialization/fitting ветки;
- строить shared identity shape для 2-3 ракурсов;
- использовать ComfyUI/segmentation для подготовки масок;
- переносить детали в material maps;
- проверять результат в browser viewer и DCC-инструментах.

## 10. Итог

Проект уже прошел путь от обзорного исследования к работающему DECA/FLAME
прототипу с WebGL-проверкой. Практическая часть не заменяет исследовательский
отчет, а усиливает его: она показывает, какие выводы остаются верными при
реальном экспорте GLB, просмотре в Three.js и сравнении coarse/detail вариантов.

Главный вывод последних экспериментов: дальнейшее качество будет расти не от
хаотичного увеличения полигонов, а от согласованного texture/mask pipeline в
правильном UV-пространстве. Следующий сильный шаг — перейти от vertex-color и
разовых визуальных проб к полноценному 2D/UV baking workflow, где маски,
текстуры и material maps становятся такими же важными артефактами, как сам
mesh.
